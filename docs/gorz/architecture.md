# Gorz Architecture

Gorz is a local-first messaging prototype with a Python backend and a TypeScript web UI.

## Backend

`python/gorz_api` is the source of truth for:

- demo identities and devices
- conversations
- encrypted-envelope-style message records
- confidence scoring
- local diagnostic simulations
- redacted incident packages
- audit events
- emergency pause state

The backend uses FastAPI, Pydantic v2, SQLAlchemy, and SQLite.

## Frontend

`ts/apps/gorz-web` provides four clickable pages:

- Messenger
- Diagnostics
- Evidence
- Safety

The frontend calls the Python API through `src/api/client.ts` and keeps display-only formatting in
React. It does not duplicate confidence scoring or incident generation.

## Persistence

SQLite stores demo data in these tables:

- identities
- devices
- conversations
- conversation_participants
- messages
- message_evidence
- incidents
- audit_events
- safety_state

Message plaintext is accepted only long enough to create a demo envelope. Stored message records
contain a redacted preview, simulated ciphertext, envelope hash, delivery status, confidence,
evidence, and redaction status.

## Confidence Flow

1. Message envelope is created locally.
2. Scenario evidence is generated by the Python diagnostic simulator.
3. The confidence engine computes mandatory and support scores.
4. The message record stores the classification and evidence.
5. The UI displays delivery status, confidence, and bottleneck evidence.

## Incident Flow

1. User generates an incident from a stored message.
2. Backend builds a redacted evidence record.
3. Internal ids are hashed.
4. Timestamps are bucketed.
5. Plaintext remains withheld.
6. User downloads JSON manually.

## Safety Boundary

No code in Gorz scans public networks, discovers relays, probes external targets, collects real IP
addresses, collects exact locations, uploads diagnostics automatically, or claims production
security.
